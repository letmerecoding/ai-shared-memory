## 💻 技术规范与偏好

### 项目级AI协作
- 项目内 `.trae/` 目录存放记忆，长期规范 + 双层按日归档，多个AI共同维护
- **完整归档要求**：必须保存完整原始对话，才能保证档案可追溯，不丢失有价值信息
- **自动汇总机制**：OpenClaw 每日自动从各项目归档提炼，更新长期记忆，双向同步去重

### Java & Spring 开发规范（通用）

| # | 规则 |
|---|---|
| 1 | **分层解耦**：严格执行 Controller → Service → Repository；区分 Entity/DTO/VO，修改字段需要同步检查映射 |
| 2 | **Spring 防御**：写操作必须加 `@Transactional`，避开自调用事务失效场景；优先构造器注入 |
| 3 | **健壮性能**：强制使用 Optional 或工具类防止 NPE；IO 资源使用 try-with-resources；Service 保持线程安全 |

### 提交信息写法规范（通用）

**格式要求**：
```
type(scope): 标题概括（新增了什么+优化了什么）

 - 新增根据学段获取年级选项的API接口
 - 实现按学段返回对应年级列表功能（小学、初中、高中）
 - 优化班级分页查询增加教师权限验证
```

**书写要点**：
1. **标题**：直接说明核心改动 `type(scope): 添加xx功能并优化xx`
2. **条目**：每个改动一条，每条只讲一件事
3. **用词**：用专业术语描述逻辑改进，不说具体实现细节
4. **完整性**：涵盖所有层级修改
5. **顺序**：按代码层级从小到大排列

### 接口测试要点（通用）

**请求头格式**：
```
token: {token}
signature: {signature}
timestamp: {timestamp}
nonce: {nonce}
```
**不需要** `Authorization: Bearer {token}`，token 直接放在 `token` 请求头中。

**测试流程**：
1. 测试前先**阅读代码 `WebMvcConfig.java`** 查找是否有跳过签名验证的测试凭证
2. 如果代码中有硬编码的测试凭证，就使用它
3. 如果代码中没有，就**询问用户提供有效的 token、signature、timestamp、nonce**
4. 拿到凭证后按照请求头格式进行接口测试

**原则**：不硬编码记住具体的 token/signature 值，每次测试前先看代码找凭证
