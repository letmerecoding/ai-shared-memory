## 🛠️ 开发方法论与设计模式

### 一、解决问题的方法论

#### 1. 问题驱动，逐步迭代

采用 **发现问题 → 定位问题 → 修复问题 → 验证效果** 的迭代思路：
- 在同步代码后，发现问题逐个修复
- 每一步都可验证，风险可控，不会因为大面积修改引入难以定位的新问题

#### 2. 多分支同步流程：从源头同步，保持分支一致性

1. 先从源分支同步功能代码
2. 然后针对当前分支的业务特性，逐个发现并修复适配问题
3. 最后补充当前分支特有的业务规则

遵循 **多分支同步差异对比流程**：
- 接口返回结构、方法签名保持和原分支一致
- 数据权限过滤保持当前分支特性
- 业务逻辑适配当前分支定位

#### 3. 数据一致性保护：预防胜于修复

在数据插入前就进行前置检查，防止冗余，避免后续清理数据的麻烦。例如：添加班级重复加入检查，查询现有关联 → 检查相同学科是否已存在 → 存在则返回提示。

#### 4. 重构优化遵循设计原则

- **单一职责原则**：将复杂逻辑提取为独立方法，每个方法只做一件事
- **消除重复代码**：统一处理流程，移除重复代码
- **代码简洁**：使用 Stream API 替代传统循环，代码更简洁
- **移除不合理限制**：移除不必要的必填验证，增加灵活性

#### 5. 缺陷修复：定位准确，最小改动

发现问题 → 定位根源 → 只改错误点，不影响其他逻辑，确保修复精准不引入新问题。

#### 6. 演进式开发：逐步逼近最优

```
尝试实现基本功能 → 发现问题 → 分析问题根源 → 修复优化 → 再重构简化 → 验证效果
```

**核心特点：**
- **小步快跑**：不一次性改完所有问题，发现一个解决一个
- **细粒度提交**：每个提交只解决一个问题，便于回滚和追踪
- **先可用后优化**：先保证功能可用，再优化代码结构
- **持续重构**：不满足于"能用"，持续优化代码结构
- **用户中心**：根据实际使用反馈调整设计，不固守初始方案

### 二、设计模式：逐步缩小查询范围

适用于 **"多表关联、条件分散"** 的查询场景。

#### 设计思路

```
1. 查询用户信息 → 得到初步筛选后的ID列表
2. 根据ID查询关联表 + 在关联表上添加过滤条件 → 进一步筛选
3. 根据过滤后的关联反向过滤主信息 → 得到最终结果列表
4. 从关联提取主键 → IN查询详细信息 → 已经是精确主键查询，不需要其他条件
```

#### 核心思想

- **谁有过滤条件谁先过滤**：过滤条件在哪个表，就在哪个表过滤
- **逐步缩小范围**：每一步都过滤掉不符合条件的数据，下一步数据量更少

### 项目级AI协作
- 项目内 `.trae/` 目录存放记忆，长期规范 + 双层按日归档，多个AI共同维护
- **完整归档要求**：必须保存完整原始对话，才能保证档案可追溯，不丢失有价值信息
- **自动汇总机制**：OpenClaw 每日自动从各项目归档提炼，更新长期记忆，双向同步去重
