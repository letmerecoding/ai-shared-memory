## 一、Java & Spring 专项规范

| # | 规则 |
|---|---|
| 1 | **分层解耦**：严格执行 Controller → Service → Repository；区分 Entity/DTO/VO，修改字段需要同步检查映射 |
| 2 | **Spring 防御**：写操作必须加 `@Transactional`，避开自调用事务失效场景；优先构造器注入 |
| 3 | **健壮性能**：强制使用 Optional 或工具类防止 NPE；IO 资源使用 try-with-resources；Service 保持线程安全 |

## 二、任务执行自检清单 (Checklist)

- [ ] **逻辑一致性**：是否扫描并处理了公共方法引用点的风险？
- [ ] **模型完整性**：Entity/DTO/VO 字段映射是否同步更新？
- [ ] **Spring 事务**：写操作是否有事务？是否存在失效场景？
- [ ] **异常与边界**：Catch 块是否包含上下文？空集合/空对象是否有防御？
- [ ] **依赖最小化**：是否引入冗余依赖或重复造轮子？
- [ ] **代码整洁**：无模糊 `//TODO`、无调试打印、无无用注释

## 三、接口测试要点

### 请求头格式
```
token: {token}
signature: {signature}
timestamp: {timestamp}
nonce: {nonce}
```
**不需要** `Authorization: Bearer {token}`，token 直接放在 `token` 请求头中。

### 测试流程
1. 测试前先**阅读代码 `WebMvcConfig.java`** 查找是否有跳过签名验证的测试凭证
2. 如果代码中有硬编码的测试凭证，就使用它
3. 如果代码中没有，就**询问用户提供有效的 token、signature、timestamp、nonce**
4. 拿到凭证后按照请求头格式进行接口测试

### 记住规则，不记住固定值
- 不硬编码记住具体的 token/signature 值
- 每次测试前先看代码找测试凭证
- 代码中没有就问用户要
- 拿到后按照格式测试

## 四、接口测试完整标准流程

| # | 步骤 | 操作要点 |
|---|------|----------|
| 1 | **查找全局鉴权配置** | - 查找 `WebMvcConfig`、`HandlerInterceptor` 等拦截器配置<br>- 分析 token 解析位置和认证规则<br>- 确认是否需要签名、时间戳、nonce 等额外参数<br>- 如果有测试环境特殊签名绕过规则，记住测试时直接使用 |
| 2 | **启动项目** | ```bash<br>mvn spring-boot:run -Dspring-boot.run.jvmArguments="-Xmx512m" &<br>```<br>- 后台运行，不阻塞终端<br>- 等待启动完成，通过 `lsof -i` 找到监听端口 |
| 3 | **构造正确的 curl 请求** | ```bash<br>curl -s -X POST \<br>  -H "Content-Type: application/json" \<br>  -H "token: {token}" \<br>  -H "timestamp: $(date +%s)" \<br>  -H "nonce: $(head -c 8 /dev/urandom | xxd -p -c 8)" \<br>  -H "signature: {signature}" \<br>  "http://localhost:{port}/{path}" \<br>  -d '{}'<br>``` |
| 4 | **分析返回结果** | - `code: 200` + `success: true` → 接口正常<br>- 根据 `data` 判断业务逻辑是否正确<br>- 空数据返回空 Map/空列表是正常业务结果 |
| 5 | **记录测试结论** | 确认：项目启动 ✅、接口路由 ✅、鉴权通过 ✅、代码逻辑 ✅、返回结构 ✅ |

## 五、多分支同步差异对比流程

| # | 步骤 | 操作要点 |
|---|------|----------|
| 1 | **明确目标分支特性** | - 先确认当前分支是什么角色（**判断依据见下表**）<br>- 理解两个分支的定位差异 |
| 2 | **查看原分支提交内容 | - `git log sourceBranch --oneline -N` 查看最近N次提交<br>- `git show commitHash` 查看每个提交具体改了什么 |
| 3 | **对比功能差异** | - 原分支新增/修改了什么功能？<br>- 当前分支原有特性是什么？<br>- 哪些功能需要保留当前分支特性？<br>- 哪些功能需要和原分支保持一致？ |
| 4 | **合并原则** | - 接口返回结构、方法签名保持和原分支一致<br>- 数据权限过滤保持当前分支特性<br>- 业务逻辑适配当前分支定位（学校管理员端看全校，教师端只看自己关联班级） |
| 5 | **验证差异** | - 合并后检查：返回结构一致 ✓、权限逻辑正确 ✓、当前分支特性保留 ✓ |

### 分支角色判断原则

**判断顺序（优先级从高到低：**
1. 第一步看 `application.properties` / `bootstrap.yml` 中的 `spring.application.name` → **实际配置优先级最高**
2. 第二步看分支名称 → 辅助确认角色定位
3. 第三步确认数据权限特性 → 选择正确的过滤逻辑

**核心原则：实际配置 > 分支命名约定**，因为配置是运行时真实生效的，命名只是约定。
