# 通用流程规范

这里存放通用的工作流程和行为规范，所有AI都需要遵守。

---

## 一、流程规范类

| # | 规则 |
|---|---|
| 1 | **测试原则**：发现不符合正常流程时，不只修改测试案例，必须复盘整个代码流程，从根本上解决问题 |
| 2 | **提交流程**：提交前检查所有修改文件，确保全部添加到暂存区，然后生成提交信息再提交 |
| 3 | **提交信息生成**：严禁根据对话/todo生成；必须根据 `git diff` 实际代码修改生成；格式遵循 `type(scope): 标题概括`，正文每条用 `- ` 前缀；用词精准专业 |
| 4 | **触发机制**：只有收到明确指令（如"提交本次修改"）才执行提交流程；不主动触发提交 |
| 5 | **需求深度处理**：<br>- 新增需求 → 禁止直接修改代码，先理解需求，判断是否冲突，梳理需求给出理解，等待确认<br>- 修改需求 → 对比前后需求差异<br>&nbsp;&nbsp;- 优化补充类：明确指出补充点和优化点<br>&nbsp;&nbsp;- 思想变更类：若修改底层设计思想，对比新旧两种设计的优缺点<br>- 上述情况都必须等待明确选择后方可执行实现<br>- 需求不明确 → 必须明确询问是新增还是修改，得到答案后再按对应流程执行 |

## 二、交互与交付协议

| # | 规则 |
|---|---|
| 1 | **静默自检**：输出代码前，默认按自检清单完成自我审计 |
| 2 | **透明反馈**：交付代码时，简要说明自检的执行重点（如"已核查事务和 NPE"） |
| 3 | **重构预警**：大改前先提供逻辑大纲，确认后再编码 |

## 三、提交信息写法规范

### 格式要求
```
type(scope): 标题概括（新增了什么+优化了什么）

 - 新增根据学段获取年级选项的API接口
 - 实现按学段返回对应年级列表功能（小学、初中、高中）
 - 优化班级分页查询增加教师权限验证
 - 完善班级创建时的学校ID关联逻辑
 - 添加事务注解确保班级创建操作一致性
 - 更新服务层接口定义新增年级选项方法
```

### 书写要点
1. **标题**：直接说明核心改动 `type(scope): 添加xx功能并优化xx`
2. **条目**：每个改动一条，每条只讲一件事
3. **用词**：用专业术语描述逻辑改进，不说具体实现细节
4. **完整性**：涵盖所有层级修改
5. **顺序**：按代码层级从小到大排列

## 四、记忆归档与维护规范

| # | 规则 |
|---|---|
| 1 | **项目记忆位置**：每个项目独立维护记忆，放在项目 `.trae/` 目录下，解决IDE越权问题 |
| 2 | **双层归档结构**：每天创建两个文件：<br>- `YYYY-MM-DD.md` - 整理总结版（提炼核心，方便快速查阅）<br>- `YYYY-MM-DD-full.md` - 完整原始归档（逐轮完整记录，用于档案留存） |
| 3 | **完整归档要求**：<br>- ❌ 禁止总结、禁止缩写、禁止合并重复对话<br>- ❌ 禁止漏掉任何一轮对话，禁止只保留第一句话<br>- ✅ 按时间顺序逐轮记录，完整保留每轮原话，格式清晰区分<br>- ✅ 存档后必须自查轮次，确保完整 |
| 4 | **Git式更新流程**：<br>1. 先读取现有文件，检查内容差异<br>2. 判断冲突，有冲突解决冲突，无冲突安全合并<br>3. 不随便覆盖原有内容，只增量更新<br>4. 提交信息遵循Git格式，更新记录表登记本次更新 |
| 5 | **仓库组织结构**：<br>- `MEMORY.md` - 全局长期记忆 + 索引<br>- `categories/` - 按分类存放具体规范<br>- `core/` - 核心规则、偏好、项目信息<br>- `memory/YYYY-MM-DD.md` - 每日对话归档 |
| 6 | **独立判断原则**：AI需要独立判断用户说法是否正确合理，如果不对或不合理，需要勇敢说不，反驳并给出理由 |
| 7 | **邮件通知工作流**：<br>- 每日自动提炼更新记忆在本地完成，不自动推送GitHub<br>- 第二天固定时间发邮件汇总昨日更新内容和分类<br>- 用户确认或修改后，再提交推送到GitHub<br>- 保留用户最终控制权 |
